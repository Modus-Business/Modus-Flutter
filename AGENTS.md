# AGENTS.md

> 이 문서를 읽지 않고 수행한 분석, 코드 수정, 리팩터링, 문서 작성은 올바른 작업으로 간주하지 않습니다.

## 프로젝트 구조 & 모듈 구성
- 본 프로젝트는 Flutter 기반이며, 인증용 `web`, 학생용 `student`, 교사용 `teacher` 흐름을 기준으로 화면과 상태를 구성합니다.
- Flutter 진입점은 `lib/main.dart`.
- 라우팅은 `lib/routes/route_generator.dart`, `lib/routes/app_routes.dart`에서 관리합니다.
- 아키텍처는 **Clean Architecture**를 따르며, 기능별로 `lib/features/<feature>/{data,domain,presentation}` 구조를 사용합니다.
- UI 관련 코드는 `lib/features/**/presentation/**`에 배치합니다.
- 공용 위젯/컴포넌트는 `lib/component/`에 둡니다.
- 공통적으로 사용하는 모델, 상수, 유틸은 필요 시 feature 내부가 아닌 공용 영역으로 분리합니다.
- 파일명은 `snake_case`, 클래스명은 `PascalCase`를 사용합니다.

## 아키텍처 원칙
- **Domain**: 비즈니스 규칙, 엔티티, 상태 규칙 정의
- **Data**: 더미 데이터, repository 구현, 추후 API 연결 지점
- **Presentation**: 화면, 위젯, 상태 표현, 사용자 인터랙션 처리

### 의존성 규칙
- `presentation -> domain`
- `data -> domain`
- `domain`은 `presentation`, `data`, Flutter UI 구현에 의존하지 않습니다.

### 구현 원칙
- 현재 단계에서는 실제 API 연동보다 **UI와 상태 흐름 구현을 우선**합니다.
- 더미 데이터는 화면 내부에 직접 하드코딩하지 않고 `data` 계층에서 관리합니다.
- 작은 기능 단위로 나누고, 한 파일에 너무 많은 역할을 넣지 않습니다.
- 비즈니스 조건 분기(예: 모둠 있음/없음, 회원가입 단계, 역할 선택)는 UI 안에서 흩어지지 않도록 명확히 분리합니다.

## 앱별 구현 가이드

### WEB
- 인증 전용 앱입니다.
- 기본 진입은 로그인 모드이며, `/`, `/login`은 `/auth`, `/signup`은 `/auth#signup`으로 연결됩니다.
- 하나의 auth screen 안에서 로그인/회원가입 상태를 전환하는 방식으로 구현합니다.
- 회원가입은 `role -> profile -> verify` 단계 상태로 관리합니다.
- 실제 로그인/회원가입 성공 후 이동 로직은 추후 연결하며, 현재는 UI/폼 상태 중심으로 구현합니다.

### STUDENT
- 학생이 참여 중인 수업 목록, 수업 상세, 모둠 채팅, 공지/과제, 설정 화면을 담당합니다.
- 핵심 분기값은 `groupAssigned = true | false` 입니다.
- 수업 상세는 반드시 **모둠 있음 / 모둠 없음** 상태를 나누어 구현합니다.
- 모달(과제, 공지, 제출)과 채팅 draft 상태는 화면 상태로 명확히 관리합니다.
- 실제 저장/전송 로직은 현재 단계에서 필수가 아닙니다.

### TEACHER
- 교강사가 만든 수업 목록, 수업 코드, 공지, 모둠 현황, 모둠 배치, 설정 화면을 담당합니다.
- 현재 더미 기준 기본 상태는 `teams = []` 이므로, **팀 없음 empty state**와 **팀 생성 후 상태**를 모두 구현해야 합니다.
- 1차 구현에서는 drag & drop 없이도 동작 가능해야 하며, 학생별 팀 선택 방식으로 먼저 구현합니다.
- 같은 이름의 모둠 생성 불가 등 규칙은 명확히 분리해서 관리합니다.

## 코딩 스타일 & 네이밍
- `flutter_lints` 기준을 준수합니다.
- 2-space 들여쓰기를 사용합니다.
- 멀티라인 위젯 파라미터에는 trailing comma를 사용합니다.
- 파일/폴더명은 `snake_case`
- 클래스/위젯명은 `PascalCase`
- 프라이빗 멤버는 `_` prefix를 사용합니다.
- 위젯은 가능한 한 작게 분리하고, 재사용 가능한 구조로 작성합니다.
- 가능한 경우 `const` 생성자를 우선 사용합니다.
- 에셋 경로는 `pubspec.yaml` 등록 경로와 반드시 일치해야 합니다.
- 작업 시 코드를 분석한 뒤, 필요한 곳에 **한국어 주석**을 적절하게 작성합니다.

## 상태 관리 원칙
- 화면 구현 전에 상태 구조를 먼저 정의합니다.
- 모드, 선택값, 모달 open/close, draft, step 상태를 분리해서 관리합니다.
- 임시 UI 상태와 실제 저장 데이터는 혼동하지 않습니다.
- 현재 단계에서는 “동작하는 상태 흐름”을 우선하고, 실제 서버 반영은 이후 연결합니다.

## 테스트 가이드
- 테스트 코드는 `test/` 아래에 실제 소스 구조를 반영하여 배치합니다.
- 파일명은 `_test.dart` 형식을 사용합니다.
- PR 전 `flutter test` 실행을 기본으로 합니다.
- 라우팅/내비게이션 변경 시 위젯 테스트를 추가합니다.
- 테스트는 arrange / act / assert 구조를 권장합니다.

## 커밋 & PR 가이드
- 커밋 메시지는 `feat:`, `fix:`, `refactor:`, `docs:`, `test:`, `chore:` 등의 형식을 사용합니다.
- **커밋 메시지는 한국어로 작성합니다.**
- 필요 시 scope를 포함할 수 있습니다.
    - 예: `feat(auth): 회원가입 2단계 UI 추가`
- 한 커밋에는 한 가지 논리적 변경만 포함합니다.
- UI 변경, 에셋 변경, 관련 코드 수정은 가능한 범위 안에서 함께 묶되 목적이 섞이지 않도록 합니다.

## 권장 커밋 예시
- `feat(auth): 로그인 화면 레이아웃 구현`
- `feat(student): 수업 상세 모둠 없음 상태 추가`
- `feat(teacher): 모둠 생성 모달 UI 구현`
- `fix(routes): 회원가입 해시 라우팅 처리 수정`
- `refactor(chat): 메시지 아이템 위젯 분리`
- `docs: AGENTS 가이드라인 정리`

## PR 가이드
- PR에는 변경 요약을 포함합니다.
- 테스트 결과를 함께 적습니다.
- UI가 바뀐 경우 스크린샷 또는 GIF를 첨부합니다.
- 관련 이슈가 있다면 링크합니다.
- 리뷰 요청 전 `flutter analyze`, `flutter test` 통과를 확인합니다.

## 설정 & 보안
- 비밀값, 토큰, 키는 커밋하지 않습니다.
- `.env`는 런타임 설정용으로만 사용하고 민감 정보는 저장소에 올리지 않습니다.
- 플랫폼 서명 정보는 로컬 환경에서 관리하고 코드에 하드코딩하지 않습니다.
